# Themed Coloring Bundles

Premade ┬Ż4.99 themed bundles (10 pages, recurring hero cast, PDF + online coloring) sold as one-time Stripe products. This doc covers what a "bundle" is end-to-end and how to launch a new one without missing a step.

## What it is

A bundle is:

1. A **HeroBundle profile** in `packages/coloring-core/src/bundles/profiles.ts` ŌĆö the static spec: heroes, page prompts, art-style block. Worker reads this when generating pages.
2. A **Bundle row** in the DB (`bundles` table) ŌĆö the runtime row: slug, name, tagline, price, page count, `published` flag, Stripe product/price IDs (test + live).
3. **10 ColoringImage rows** with `bundleId` set + ordered by `bundleSequence` (the actual coloring pages, generated by the worker).
4. **5 listing images** on R2 at `bundles/{slug}/listings/` ŌĆö Hero + BrandCard + 3 PageGrids (Satori-rendered marketing images, used on the product page carousel + Stripe product photos).
5. **Hero reference images** on R2 at `bundles/{slug}/hero-refs/{heroId}.png` ŌĆö line-art "model sheets" the page generator conditions on so every page draws the same heroes consistently.
6. **Brand character** on R2 at `bundles/{slug}/brand-character.png` ŌĆö polished colored hero used on the BrandCard listing image.
7. A **Stripe Product + Price** in both test and live mode (separate IDs stored on the Bundle row).

## Brainstorming a new bundle theme

Before launching, you need a theme + cast. To research fresh themes against current Etsy / Amazon / Pinterest trends and auto-scaffold the profile entry, run:

```bash
cd apps/chunky-crayon-web
pnpm tsx -r dotenv/config scripts/research-bundle-ideas.ts \
  dotenv_config_path=.env.local

# Bias the proposals toward a vibe / season / audience hint:
... --hint="cozy autumn, animal characters, ages 4-7"

# Print only ŌĆö don't auto-write to profiles.ts:
... --no-write
```

The flow:

1. Reads existing `HERO_BUNDLES` so it tells Perplexity what NOT to repeat
2. Perplexity Sonar proposes 3 candidate themes with rationale + 4 hero sketches + 10 page ideas each
3. You pick `1`, `2`, or `3` (or `skip`)
4. Claude flesh-out: turns the picked sketch into a polished profile (4-6 signatureDetails + ~150 word referenceSheetPrompt + funFact per hero, full pagePrompts, pageCast)
5. You confirm `y/N`
6. On `y`: appends the new `HeroBundle` export to `profiles.ts` AND adds it to the `HERO_BUNDLES` record AND adds it to `coloring-core/src/index.ts` exports

After auto-write, sanity-check the heroes (referenceSheetPrompts especially ŌĆö these condition every page), then run the launcher in **Step 2** below. The runbook continues from there.

Cost: ~$0.10 per research run. Env vars: `PERPLEXITY_API_KEY` + `ANTHROPIC_API_KEY`.

## Launching a new bundle

### Step 1: Define the profile

Edit `packages/coloring-core/src/bundles/profiles.ts`. Copy the `DINO_DANCE_PARTY` block as a template. The four launch bundles already have their `pagePrompts` filled in ŌĆö you only need to write the `heroes` array (3-4 characters), `signatureDetails`, `referenceSheetPrompt`, and `funFact`.

Hero requirements:

- **id**: kebab-case, unique within the bundle (e.g. `unicorn-stardust`)
- **name**: friendly display name (e.g. "Stardust the Unicorn")
- **signatureDetails**: comma-separated visual anchors that MUST appear on every page (e.g. "horn shaped like a candy cane, star-pattern hooves, mane braided with rainbow ribbons"). The page generator uses these as constraints; the QA gate checks for them.
- **referenceSheetPrompt**: a single-paragraph prompt for gpt-image-2 to draw the model sheet. Aim for ~150-200 words; describe pose (front three-quarter view), background (plain white), line weight, no shading. See existing entries for tone.
- **funFact**: 1 sentence shown on the BrandCard listing image.

Then add an export for the new bundle in `packages/coloring-core/src/index.ts`.

### Step 2: Run the launcher

From `apps/chunky-crayon-web`:

```bash
# Dry run first ŌĆö no API calls, no R2 writes, no DB writes:
pnpm tsx -r dotenv/config scripts/launch-bundle.ts \
  --slug=unicorn-rainbow-rally \
  --name="Unicorn Rainbow Rally" \
  --tagline="A wild unicorn race across the sky to bring back the missing rainbow colors" \
  --dry \
  dotenv_config_path=.env.local

# When the dry run looks right, drop --dry and run for real:
pnpm tsx -r dotenv/config scripts/launch-bundle.ts \
  --slug=unicorn-rainbow-rally \
  --name="Unicorn Rainbow Rally" \
  --tagline="A wild unicorn race across the sky to bring back the missing rainbow colors" \
  dotenv_config_path=.env.local
```

The orchestrator runs nine steps in sequence:

| #   | Step                          | Time   | Cost   | Idempotent                 |
| --- | ----------------------------- | ------ | ------ | -------------------------- |
| 1   | Verify profile + heroes       | <1s    | free   | yes                        |
| 2   | Generate hero refs            | ~15min | ~$0.16 | skip-if-on-r2              |
| 3   | Bg-remove hero refs           | ~1min  | ~$0.00 | skip-if-on-r2              |
| 4   | Generate brand character      | ~3min  | ~$0.08 | skip-if-on-r2              |
| 5   | Bg-remove brand character     | ~30s   | ~$0.00 | skip-if-on-r2              |
| 6   | Seed Bundle row               | <1s    | free   | upsert                     |
| 7   | Generate 10 pages via worker  | ~35min | ~$1.50 | skip-if-rows-already-ready |
| 8   | Render 5 listing images       | ~2min  | free   | overwrite                  |
| 9   | Create Stripe product + price | <5s    | free   | check-existing-then-create |

Total wall-clock: ~50min. Total cost: ~$1.75.

Re-running a partial launch picks up where it left off ŌĆö every step checks if its output already exists before doing work.

Useful flags:

- `--dry` ŌĆö print what would happen without doing it. Always run this first.
- `--skip-pages` ŌĆö skip the long page-gen step. Useful when iterating on listings only.
- `--skip-stripe` ŌĆö skip Stripe product creation. Useful when product already exists in dashboard and you just want to refresh listings.
- `--page-count=12 --price-pence=799 --currency=usd` ŌĆö override seed defaults. Defaults: profile.pagePrompts.length / 499 / gbp.

### Step 3: Verify before publishing

The orchestrator leaves `published: false`. Before flipping it true:

1. **Visit the product page in dev**: `http://localhost:3000/en/products/digital/{slug}`. Carousel should show all 5 listing images, page grid at bottom should show all 10 thumbnails.
2. **Check the listing images**: `http://localhost:3000/en/dev/listing-templates?slug={slug}` ŌĆö preview tool side by side.
3. **Check Stripe dashboard**: product exists in test mode, price is correct, images attached.
4. **Test purchase end-to-end** in test mode: buy ŌåÆ webhook fulfils ŌåÆ email sends ŌåÆ thank-you page shows download button ŌåÆ tokened download works.

When happy, flip `published`:

```sql
UPDATE bundles SET published = true WHERE slug = 'unicorn-rainbow-rally';
```

Run this against **dev** first (visit on localhost), then **prod**.

### Step 4: Live-mode Stripe

The orchestrator uses whatever `STRIPE_SECRET_KEY` is in `.env.local` ŌĆö usually test mode. To create the live-mode product, swap to live keys and re-run with `--skip-pages` (pages already exist) and a fresh slug check:

```bash
STRIPE_SECRET_KEY=sk_live_... pnpm tsx -r dotenv/config scripts/create-stripe-bundle-product.ts \
  --slug=unicorn-rainbow-rally \
  --no-checkout \
  dotenv_config_path=.env.local
```

The Bundle row stores both `stripeProductId` (live) and `stripeProductIdTest` (test); the checkout flow picks the right one based on which Stripe key is active at runtime.

## Failure modes (real ones we've hit)

| Symptom                                                                         | Root cause                                                                | Fix                                                                                                          |
| ------------------------------------------------------------------------------- | ------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| `bundle_purchases_bundleId_fkey` violation on Stripe webhook                    | `.env.local` pointing at prod Neon when buyer test-purchased on localhost | Re-pull dev env from Vercel ŌĆö never sed-fix DATABASE_URL                                                     |
| Listing images render but coloring pages all show same hero face                | Hero ref on R2 was generated before signatureDetails were finalised       | `--force` regenerate hero refs, then re-run worker page gen                                                  |
| `EmptyGenerateStaticParamsError` on Vercel build for `/products/digital/[slug]` | New bundle in dev DB but prod DB has no rows yet                          | Either seed prod row first OR delete `generateStaticParams` and use `await connection()` until prod has data |
| Worker page-gen times out after 60min                                           | Gemini outage or QA gate rejecting every retry                            | Check worker logs at the box; manually accept stuck pages via `accept-bundle-page.ts`                        |
| Listing images on Stripe product look stretched                                 | Stripe prefers square; we upload 1200├Ś1500 JPGs                           | Stripe handles cropping fine ŌĆö don't try to preempt with extra resizing                                      |
| Brand character has visible white halo around edges                             | Replicate bg-remover threshold too aggressive                             | Re-run `remove-bundle-character-background.ts` with `--force`; iterate on prompt to push tighter outline     |
| Hero refs cost more than expected                                               | Used `--force` on a 4-hero bundle                                         | Don't `--force` unless you actually changed the prompt; idempotent skip is the default for a reason          |
| Test purchase succeeded but no email                                            | Mailtrap creds in `.env.local` stale                                      | New CC Mailtrap project ŌĆö pull fresh creds, update Vercel dev+preview envs too                               |

## Why these scripts exist as scripts (not server actions)

The full launch involves OpenAI, Replicate, R2, the worker (Hetzner box), Stripe, and the DB ŌĆö running it through a server action would either time out (Vercel function limits) or require the user to keep a browser tab open for ~50min. Scripts run from a developer's machine against prod env vars + Hetzner endpoints, take as long as they need, and emit progress to stdout.

The orchestrator (`launch-bundle.ts`) calls some of these directly (hero-refs, seed-bundle) by importing their exported functions, and shells out to others (`remove-*`, `generate-bundle-brand-character`, `create-stripe-bundle-product`) via `child_process` since they were built as standalone CLIs first. Each shelled script streams its output through to the orchestrator's stdout naturally.
