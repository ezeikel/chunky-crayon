# Themed Coloring Bundles

Premade £4.99 themed bundles (10 pages, recurring hero cast, PDF + online coloring) sold as one-time Stripe products. This doc covers what a "bundle" is end-to-end and how to launch a new one without missing a step.

## What it is

A bundle is:

1. A **HeroBundle profile** in `packages/coloring-core/src/bundles/profiles.ts` — the static spec: heroes, page prompts, art-style block. Worker reads this when generating pages.
2. A **Bundle row** in the DB (`bundles` table) — the runtime row: slug, name, tagline, price, page count, `published` flag, Stripe product/price IDs (test + live).
3. **10 ColoringImage rows** with `bundleId` set + ordered by `bundleSequence` (the actual coloring pages, generated by the worker).
4. **5 listing images** on R2 at `bundles/{slug}/listings/` — Hero + BrandCard + 3 PageGrids (Satori-rendered marketing images, used on the product page carousel + Stripe product photos).
5. **Hero reference images** on R2 at `bundles/{slug}/hero-refs/{heroId}.png` — line-art "model sheets" the page generator conditions on so every page draws the same heroes consistently.
6. **Brand character** on R2 at `bundles/{slug}/brand-character.png` — polished colored hero used on the BrandCard listing image.
7. A **Stripe Product + Price** in both test and live mode (separate IDs stored on the Bundle row).

## Architecture

```
profile (coloring-core)
       │
       │  ──► hero-refs (gpt-image-2)        ──► R2 hero-refs/{id}.png
       │  ──► hero-refs bg-removed           ──► R2 hero-refs/{id}-transparent.png
       │  ──► brand-character (gpt-image-2)  ──► R2 brand-character.png
       │  ──► brand-character bg-removed     ──► R2 brand-character.png (RGBA)
       │
       ▼
seed-bundle (DB)  ──► bundles row, published=false
       │
       ▼
worker /generate/bundle-all-pages
       │  ──► Gemini Nano Banana × 10 pages, conditioned on hero refs
       │  ──► QA gate (vision) per page
       │  ──► retries until pass or budget exhausted
       │
       ▼  10 ColoringImage rows, status=READY, bundleId set
listing builds (worker scripts)
       │  ──► Hero JPG, BrandCard JPG, PageGrid 1/2/3 JPGs
       │  ──► R2 bundles/{slug}/listings/*.jpg
       │
       ▼
create-stripe-bundle-product
       │  ──► Stripe Product + Price (live OR test depending on STRIPE_SECRET)
       │  ──► writes IDs back to bundles row
       │
       ▼
flip published=true ──► appears on /products/digital + carousel
```

## Brainstorming a new bundle theme

Before launching, you need a theme + cast. To research fresh themes against current Etsy / Amazon / Pinterest trends and auto-scaffold the profile entry, run:

```bash
cd apps/chunky-crayon-web
pnpm tsx -r dotenv/config scripts/research-bundle-ideas.ts \
  dotenv_config_path=.env.local

# Bias the proposals toward a vibe / season / audience hint:
... --hint="cozy autumn, animal characters, ages 4-7"

# Print only — don't auto-write to profiles.ts:
... --no-write
```

The flow:

1. Reads existing `HERO_BUNDLES` so it tells Perplexity what NOT to repeat
2. Perplexity Sonar proposes 3 candidate themes with rationale + 4 hero sketches + 10 page ideas each
3. You pick `1`, `2`, or `3` (or `skip`)
4. Claude flesh-out: turns the picked sketch into a polished profile (4-6 signatureDetails + ~150 word referenceSheetPrompt + funFact per hero, full pagePrompts, pageCast)
5. You confirm `y/N`
6. On `y`: appends the new `HeroBundle` export to `profiles.ts` AND adds it to the `HERO_BUNDLES` record AND adds it to `coloring-core/src/index.ts` exports

After auto-write, sanity-check the heroes (referenceSheetPrompts especially — these condition every page), then run the launcher in **Step 2** below. The runbook continues from there.

Cost: ~$0.10 per research run. Env vars: `PERPLEXITY_API_KEY` + `ANTHROPIC_API_KEY`.

### Restoring a previous flesh-out (--from-dump)

Every flesh-out call dumps its raw response to `apps/chunky-crayon-web/scripts/out/flesh-out-{slug}.json` _before_ parsing. This means:

- A failed parse (Opus wrapping quirk, schema drift) doesn't lose the response
- A `--no-write` dry-run that produces a great profile can be replayed into the codebase later without re-burning Sonar + Claude calls

To replay a saved flesh-out into `profiles.ts`:

```bash
pnpm tsx -r dotenv/config scripts/research-bundle-ideas.ts \
  --from-dump=scripts/out/flesh-out-bakery-buddy-bakers.json \
  --slug=bakery-buddy-bakers \
  --name="Bakery Buddy Bakers" \
  --tagline="Mix, bake, and decorate with four adorable bakers" \
  dotenv_config_path=.env.local
```

The `--slug` / `--name` / `--tagline` flags are required because the dump only stores the fleshed profile (heroes + pageCast + pagePrompts), not the candidate metadata. Same `y/N` confirm + same auto-write to `profiles.ts` and `index.ts`. Zero API calls.

The four stubs already have `pagePrompts` written — the prompts reference specific characters by description ("a baby unicorn with a swirly horn"). Two paths to finish a stub:

1. **Hand-write heroes** in `profiles.ts` to match the existing prompts. Use `DINO_DANCE_PARTY` as the shape reference. Keeps the existing prompts intact.
2. **Re-run `research-bundle-ideas.ts` with a hint** like `--hint="unicorns racing across the sky to bring back lost rainbow colors, cast of 4"`, pick the matching candidate, auto-write. Will overwrite the existing `pagePrompts` with fresh ones.

Path 1 is faster and honors the in-flight prompts. Path 2 is easier when the existing prompts no longer match where you want the bundle to go.

`launch-bundle.ts` rejects bundles with empty `heroes[]` at Step 1 — fill them before running.

## Launching a new bundle

### Step 1: Define the profile

Edit `packages/coloring-core/src/bundles/profiles.ts`. Copy the `DINO_DANCE_PARTY` block as a template. The four launch bundles already have their `pagePrompts` filled in — you only need to write the `heroes` array (3-4 characters), `signatureDetails`, `referenceSheetPrompt`, and `funFact`.

Hero requirements:

- **id**: kebab-case, unique within the bundle (e.g. `unicorn-stardust`)
- **name**: friendly display name (e.g. "Stardust the Unicorn")
- **signatureDetails**: comma-separated visual anchors that MUST appear on every page (e.g. "horn shaped like a candy cane, star-pattern hooves, mane braided with rainbow ribbons"). The page generator uses these as constraints; the QA gate checks for them.
- **referenceSheetPrompt**: a single-paragraph prompt for gpt-image-2 to draw the model sheet. Aim for ~150-200 words; describe pose (front three-quarter view), background (plain white), line weight, no shading. See existing entries for tone.
- **funFact**: 1 sentence shown on the BrandCard listing image.

Then add an export for the new bundle in `packages/coloring-core/src/index.ts`.

### Step 2: Run the launcher

From `apps/chunky-crayon-web`:

```bash
# Dry run first — no API calls, no R2 writes, no DB writes:
pnpm tsx -r dotenv/config scripts/launch-bundle.ts \
  --slug=unicorn-rainbow-rally \
  --name="Unicorn Rainbow Rally" \
  --tagline="A wild unicorn race across the sky to bring back the missing rainbow colors" \
  --dry \
  dotenv_config_path=.env.local

# When the dry run looks right, drop --dry and run for real:
pnpm tsx -r dotenv/config scripts/launch-bundle.ts \
  --slug=unicorn-rainbow-rally \
  --name="Unicorn Rainbow Rally" \
  --tagline="A wild unicorn race across the sky to bring back the missing rainbow colors" \
  dotenv_config_path=.env.local
```

The orchestrator runs nine steps in sequence:

| #   | Step                          | Time   | Cost   | Idempotent                 |
| --- | ----------------------------- | ------ | ------ | -------------------------- |
| 1   | Verify profile + heroes       | <1s    | free   | yes                        |
| 2   | Generate hero refs            | ~15min | ~$0.16 | skip-if-on-r2              |
| 3   | Bg-remove hero refs           | ~1min  | ~$0.00 | skip-if-on-r2              |
| 4   | Generate brand character      | ~3min  | ~$0.08 | skip-if-on-r2              |
| 5   | Bg-remove brand character     | ~30s   | ~$0.00 | skip-if-on-r2              |
| 6   | Seed Bundle row               | <1s    | free   | upsert                     |
| 7   | Generate 10 pages via worker  | ~35min | ~$1.50 | skip-if-rows-already-ready |
| 8   | Render 5 listing images       | ~2min  | free   | overwrite                  |
| 9   | Create Stripe product + price | <5s    | free   | check-existing-then-create |

Total wall-clock: ~50min. Total cost: ~$1.75.

Re-running a partial launch picks up where it left off — every step checks if its output already exists before doing work.

Useful flags:

- `--dry` — print what would happen without doing it. Always run this first.
- `--skip-pages` — skip the long page-gen step. Useful when iterating on listings only.
- `--skip-stripe` — skip Stripe product creation. Useful when product already exists in dashboard and you just want to refresh listings.
- `--page-count=12 --price-pence=799 --currency=usd` — override seed defaults. Defaults: profile.pagePrompts.length / 499 / gbp.

### Step 3: Verify before publishing

The orchestrator leaves `published: false`. Before flipping it true:

1. **Visit the product page in dev**: `http://localhost:3000/en/products/digital/{slug}`. Carousel should show all 5 listing images, page grid at bottom should show all 10 thumbnails.
2. **Check the listing images**: `http://localhost:3000/en/dev/listing-templates?slug={slug}` — preview tool side by side.
3. **Check Stripe dashboard**: product exists in test mode, price is correct, images attached.
4. **Test purchase end-to-end** in test mode: buy → webhook fulfils → email sends → thank-you page shows download button → tokened download works.

When happy, flip `published`:

```sql
UPDATE bundles SET published = true WHERE slug = 'unicorn-rainbow-rally';
```

Run this against **dev** first (visit on localhost), then **prod**.

### Step 4: Promote dev → prod

Once a bundle is verified in dev, promote it to prod with `promote-bundle.ts`. This script copies the entire dev R2 prefix to prod, mirrors the Bundle + ColoringImage rows into prod DB, and creates the live Stripe product in one shot.

```bash
cd apps/chunky-crayon-web

# Pull a prod env file (one-time per promotion session):
vercel env pull .env.production.local --environment=production

# Dry-run first:
pnpm tsx -r dotenv/config scripts/promote-bundle.ts \
  --slug=bakery-buddy-bakers \
  --prod-env-file=.env.production.local \
  --dry \
  dotenv_config_path=.env.local

# When dry looks right, drop --dry:
pnpm tsx -r dotenv/config scripts/promote-bundle.ts \
  --slug=bakery-buddy-bakers \
  --prod-env-file=.env.production.local \
  dotenv_config_path=.env.local
```

What it does:

1. Reads dev Bundle row + its 10 ColoringImage rows
2. Copies every R2 object under `bundles/{slug}/` from dev bucket to prod bucket (listings, hero-refs, brand-character, all 10 pages' webp + svg). Skip-if-exists.
3. Copies per-image assets that live outside the bundle prefix (regionMap, qrCode, coloredReference) by URL. Skipped silently if absent.
4. Upserts prod Bundle row with hostnames rewritten from dev R2_PUBLIC_URL to prod R2_PUBLIC_URL. Always starts as `published: false`.
5. Upserts each ColoringImage row in prod (preserves `id`s — same uuid in dev and prod for a given bundle page).
6. Creates / updates Stripe product in **live mode** using `STRIPE_SECRET_KEY` from the prod env file. Writes live IDs to prod Bundle row.

Guards: refuses to run if dev and prod R2 buckets or DATABASE_URLs are identical.

After promotion, prod has the bundle in DRAFT. To go live:

```sql
UPDATE bundles SET published = true WHERE slug = 'bakery-buddy-bakers';
```

`--skip-stripe` skips just the Stripe step (e.g. when re-running to refresh listings without touching the product). `--dry` runs the whole flow with no R2 writes / DB writes / Stripe calls.

#### Region-store backfill (Magic Brush)

Bundle page generation now triggers region-store + colored-reference + fill-points generation as part of the persist step (fire-and-forget). For pages generated before that change, backfill them:

```bash
# Just the bundle in question:
pnpm tsx -r dotenv/config scripts/backfill-region-stores.ts \
  --bundle=dino-dance-party \
  dotenv_config_path=.env.local

# Or specifically prod for an already-promoted bundle (use prod env file):
DATABASE_URL=<prod-url> R2_BUCKET=<prod-bucket> ... \
  pnpm tsx scripts/backfill-region-stores.ts --bundle=dino-dance-party
```

Without region maps, Magic Brush silently falls back to legacy `fillPointsJson` (which usually exists) — pages still load, just without per-pixel region accuracy on the magic brush.

### Step 5: Live-mode Stripe (legacy single-step path)

The orchestrator uses whatever `STRIPE_SECRET_KEY` is in `.env.local` — usually test mode. To create the live-mode product, swap to live keys and re-run with `--skip-pages` (pages already exist) and a fresh slug check:

```bash
STRIPE_SECRET_KEY=sk_live_... pnpm tsx -r dotenv/config scripts/create-stripe-bundle-product.ts \
  --slug=unicorn-rainbow-rally \
  --no-checkout \
  dotenv_config_path=.env.local
```

The Bundle row stores both `stripeProductId` (live) and `stripeProductIdTest` (test); the checkout flow picks the right one based on which Stripe key is active at runtime.

## Failure modes (real ones we've hit)

| Symptom                                                                         | Root cause                                                                | Fix                                                                                                          |
| ------------------------------------------------------------------------------- | ------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| `bundle_purchases_bundleId_fkey` violation on Stripe webhook                    | `.env.local` pointing at prod Neon when buyer test-purchased on localhost | Re-pull dev env from Vercel — never sed-fix DATABASE_URL                                                     |
| Listing images render but coloring pages all show same hero face                | Hero ref on R2 was generated before signatureDetails were finalised       | `--force` regenerate hero refs, then re-run worker page gen                                                  |
| `EmptyGenerateStaticParamsError` on Vercel build for `/products/digital/[slug]` | New bundle in dev DB but prod DB has no rows yet                          | Either seed prod row first OR delete `generateStaticParams` and use `await connection()` until prod has data |
| Worker page-gen times out after 60min                                           | Gemini outage or QA gate rejecting every retry                            | Check worker logs at the box; manually accept stuck pages via `accept-bundle-page.ts`                        |
| Listing images on Stripe product look stretched                                 | Stripe prefers square; we upload 1200×1500 JPGs                           | Stripe handles cropping fine — don't try to preempt with extra resizing                                      |
| Brand character has visible white halo around edges                             | Replicate bg-remover threshold too aggressive                             | Re-run `remove-bundle-character-background.ts` with `--force`; iterate on prompt to push tighter outline     |
| Hero refs cost more than expected                                               | Used `--force` on a 4-hero bundle                                         | Don't `--force` unless you actually changed the prompt; idempotent skip is the default for a reason          |
| Test purchase succeeded but no email                                            | Mailtrap creds in `.env.local` stale                                      | New CC Mailtrap project — pull fresh creds, update Vercel dev+preview envs too                               |

## Why these scripts exist as scripts (not server actions)

The full launch involves OpenAI, Replicate, R2, the worker (Hetzner box), Stripe, and the DB — running it through a server action would either time out (Vercel function limits) or require the user to keep a browser tab open for ~50min. Scripts run from a developer's machine against prod env vars + Hetzner endpoints, take as long as they need, and emit progress to stdout.

The orchestrator (`launch-bundle.ts`) calls some of these directly (hero-refs, seed-bundle) by importing their exported functions, and shells out to others (`remove-*`, `generate-bundle-brand-character`, `create-stripe-bundle-product`) via `child_process` since they were built as standalone CLIs first. Each shelled script streams its output through to the orchestrator's stdout naturally.
